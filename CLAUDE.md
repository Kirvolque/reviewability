# CLAUDE.md

## Project Overview

`reviewability` is a Python tool that analyzes code diffs and computes
**Reviewability Scores** — metrics that approximate how hard a diff is to review.
Scores range from 0.0 (hardest) to 1.0 (easiest). Designed for use in CI/CD pipelines.

## Commands

```bash
source .venv/bin/activate

# Run tests
pytest tests/ -v

# Lint
ruff check .

# Format
ruff format .

# Run CLI
reviewability HEAD~1 HEAD
git diff HEAD~1 | reviewability --from-stdin
```

## Architecture

```
src/reviewability/
  domain/
    models.py       — Diff, FileDiff, Hunk (raw diff data)
    report.py       — AnalysisReport, OverallAnalysis, FileAnalysis, HunkAnalysis,
                      MetricValue, MetricValueType
  metrics/
    base.py         — Metric, HunkMetric, FileMetric, OverallMetric (interfaces)
    builtin.py      — Built-in metric implementations
    registry.py     — MetricRegistry (the only mutable structure)
  rules/
    engine.py       — Rule, RuleViolation, RuleEngine
  parser/
    git.py          — parse_diff_text(), parse_git_diff() via unidiff
  cli.py            — CLI entry point

tests/
  fixtures/         — Real .diff files generated by generate.sh
  functional/       — Tests using fixture diffs, compare full parsed objects
  unit/             — Tests using in-code Diff/Hunk construction
```

## Domain Model

```
Diff
 └── FileDiff  (path, is_new, is_deleted, old_path if renamed)
      └── Hunk  (source/target coords, added/removed/context lines)
```

A **hunk** is a contiguous block of changes within a single file.

## Metric System

Three levels, calculated in order:

1. **HunkMetric** — `calculate(hunk: Hunk) -> MetricValue`
2. **FileMetric** — `calculate(file: FileDiff) -> MetricValue`
3. **OverallMetric** — `calculate(hunks: list[HunkAnalysis], files: list[FileAnalysis]) -> MetricValue`

Overall metrics are **second-order**: they derive from already-computed analyses,
not from the raw diff.

To add a new metric: subclass the appropriate base, define the four class attributes
(`name`, `value_type`, `description`, `remediation`), implement `calculate()`,
and register it via `registry.add()`.

## Report Structure

```
AnalysisReport
  ├── overall: OverallAnalysis   (metrics + score)
  ├── files:   list[FileAnalysis]  (per file: metrics + score)
  └── hunks:   list[HunkAnalysis]  (per hunk: metrics + score)
```

Scores are `float` — always present, populated by a Scorer (not yet implemented,
currently `0.0` placeholder).

## Coding Principles

- **Immutability**: all dataclasses are `frozen=True`. Objects are fully constructed
  in one shot — never mutated after creation. Avoid `dataclasses.replace()`.
- **Bottom-up construction**: children are fully built before their parent.
  Scores are computed before constructing the analysis object, not patched in afterwards.
- **Effectively final**: variables are not reassigned after initial assignment.
  Collections are built via comprehensions, never via `.append()` loops.
- **MetricRegistry** is the only intentionally mutable structure — add-only.
- All public interfaces and class fields must have type annotations.
- All public interfaces must have docstrings.

## Tests

Functional tests use real `.diff` files from `tests/fixtures/`.
To regenerate fixtures: `bash tests/fixtures/generate.sh`

Tests compare full parsed objects against exact expected values — not just
individual field assertions.

## Tooling

- Python 3.13
- `unidiff` for diff parsing
- `pytest` for tests
- `ruff` for linting and formatting (`line-length = 100`, rules: E, F, I)
